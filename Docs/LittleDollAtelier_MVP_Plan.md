# Little Doll Atelier: MVP開発計画書

「Little Doll Atelier: My Little Alchemist」のコンセプトに基づき、ソロ開発で最初の1週間で到達可能な「MVP（Minimum Viable Product）」としてスコープを絞り込んだ開発計画を提案します。

## 1. MVPに含める機能（1週間での必須スコープ）
まずは「ポモドーロで作業する → 報酬を得る → 人形が反応する（成長・魔法の進捗）」という**コアゲームループの最小単位**が回ることを目標にします。

* **ホーム画面（工房）**: 人形（最初は1体のみ・仮画像可）の表示と、所持アイテム・お金のUI表示
* **ポモドーロタイマー（基礎）**: 作業時間/休憩時間のカウントダウン、一時停止/終了機能、バックグラウンド（スリープ）復帰時の時間経過計算
* **簡易タスク管理**: 1セッションごとのテキスト入力（「○○の作業をする」等）と完了チェック
* **報酬システム**: タイマー完了時に固定またはランダムで「お金」と「基本素材（2〜3種）」を付与
* **簡易魔法研究（クラフト）**: 獲得したお金・素材を消費して「魔法の研究進捗（ゲージ等）」を進める機能
* **セーブ＆ロード**: 現在のお金、素材所持数、魔法の進捗、人形の簡易ステータスのローカル保存（JSON等）

## 2. 後回しにする機能（MVP以降に実装）
最初の1週間ではモチベーション低下や実装の複雑化を避けるため、以下は**あえて実装しません（またはモック・仕様定義のみに留める）**。

* 複数人形の作成・切り替えシステム（MVPでは1体固定）
* 詳細な着せ替え・家具配置システムのUI実装
* 複雑なMood（気分）や性格による分岐ロジック（MVPではランダムなテキスト表示程度に留める）
* 楽曲のアンロックと細かな再生システム（MVPでは1曲ループか、解放されましたというUI表示のみ）
* ログインボーナス、デイリー/ウィークリーミッション
* 街のショップ機能（MVPはクラフト画面内で完結させる）
* キャラ同士の交流

## 3. 画面一覧とナビゲーションフロー
MVPでは画面遷移（シーン切り替え）を極力減らし、単一シーン内のUIポップアップやパネル切り替えで構築します。

1. **タイトル画面**
   * 「Start」ボタンのみ。
2. **ホーム画面（メイン画面）**
   * 中央に人形の立ち絵（または3Dモデル）。タップでランダムな台詞吹き出しが出る。
   * 上部に所持金・素材アイコンリスト。
   * 下部に「Timer」「Magic(Craft)」「Doll(ステータス確認用)」のナビゲーションボタン。
3. **Timer（ポモドーロ）パネル**
   * 今回のタスク入力欄（1行テキスト）。
   * 25分/5分の切り替え、Start/Give Upボタン。
   * カウントダウン円形プログレスバー。
   * 完了時のリザルトポップアップ（「○G、素材Aを獲得！」）。
4. **Magic（魔法研究）パネル**
   * 研究可能な魔法リスト（MVPではダミーデータ含め3つ程度）。
   * 必要素材と「Research」ボタン。メーターが貯まるか、1発で完成する。

## 4. データ構造（コアデータモデル）
拡張性を考慮しつつ、MVP段階で必要なデータ構造です。

### マスタデータ (ScriptableObjectで実装)
* `MaterialData`: 素材ID、名前、アイコン画像、レアリティ
* `MagicRecipeData`: 魔法ID、名前、必要素材リスト（IDと数）、必要ゴールド、解放されるもの（楽曲ID等）

### セーブデータ (JSON等でシリアライズ)
* `PlayerSaveData`:
  * `int money`
  * `Dictionary<string, int> inventory` (素材ID -> 所持数)
  * `List<string> unlockedMagics` (解放済み魔法IDリスト)
* `DollSaveData`:
  * `string dollName`
  * `PersonalityStats personality` (curiosity, diligence 等の int値)
  * `RelationshipStats relationship` (trust, affection 等の int値)
  * `int currentMood` (enum化)

### セッションデータ (メモリ上のみ保存されないデータ)
* `TimerState`: 現在の状態（Work, Break, Idle）、残り時間、設定時間、開始時刻（バックグラウンド計算用）
* `CurrentTask`: 入力されたタスク文字列

## 5. 実装タスクと優先順位（1週間のマイルストーン）
ソロ開発者が迷わず進めるためのクリーンな開発順序です。

### 【Day 1: 基盤とデータ構築】
* Unityプロジェクト作成（URP推奨）
* 画面UIの基本レイアウト作成（Canvas、ダミーのパネルとボタン配置）
* マスタデータ（ScriptableObject）のスキーマ定義とテストデータ投入
* セーブ/ロードシステムの基盤実装（一旦JSONベタ書きでOK）

### 【Day 2: コア機能「ポモドーロタイマー」の実装】*※最大のリスク箇所*
* タイマーのカウントダウン処理（`Time.unscaledDeltaTime` または `async/await` 利用）
* バックグラウンド（スマホのスリープやPC別タブ操作等）時の時間経過補償ロジックの作成
* タイマーのUI反映

### 【Day 3: 報酬・インベントリサイクルの実装】
* タイマー完了時のコールバック発火
* 簡易タスク入力と完了検知
* インベントリ（所持金・素材）の加算処理と上部UI更新
* セーブデータへの書き込み処理連携

### 【Day 4: 魔法研究・クラフト（消費サイクル）の実装】
* Magic（魔法研究）パネルのUI実装
* 素材枯渇時のボタン非活性化など、条件判定ロジック
* 素材消費とアンロック状態の保存

### 【Day 5: お人形との簡易インタラクション】
* ホーム画面の人形表示処理（仮スプライト等）
* ドールタップ時の吹き出しUI表示
* 簡易なランダム台詞出力ロジック（性格やMoodのデータだけは読み込んでおく）

### 【Day 6: 手触りの向上（モック・仮素材の適用）】
* 各種ボタンの簡単なSE（クリック音など）追加
* タイマー完了時、魔法完成時の簡単なパーティクル（VFX）実装
* Cozy感を出すための全体的なカラーパレット（色使い）の調整

### 【Day 7: デバッグと通しプレイ】
* 実際に25分回してみてバグがないか確認
* セーブデータが壊れないか、アプリ再起動時にタイマーが破綻しないかのテスト
* リファクタリング

## 6. プロジェクトにおける最大のリスクと対策
1. **タイマーのバックグラウンド動作の破綻**
   * **対策**: 厳密なリアルタイム性をUnity側に求めない。アプリが `OnApplicationPause` または `OnApplicationFocus` で裏に回った瞬間の システム時刻(`DateTime.Now`) を保存し、復帰時に現在時刻と比較して経過時間を算出し、タイマーをワープさせる手法（放置ゲームでよく使われる仕組み）を最初に必ず実装・検証します。
2. **"Cozy&Cute"なアセット制作に時間を奪われる**
   * **対策**: MVP期間中は徹底して「仮素材（図形やフリー素材）」を使います。UIもまずはUnity標準のボタンで組み、ロジックが完成してから一気にスプライトを差し替える方針にします。ゲーム画面の見た目がしょぼくても焦らないことが大切です。
3. **着せ替えなどの拡張でデータ構造が崩壊する**
   * **対策**: 人形の見た目関連は早い段階で「パーツごとに別れたPrefab（またはSpriteのレイヤー）」構成になることを想定します。MVP段階の1枚絵による仮実装であっても、人形本体のオブジェクトには「髪」「服」などのアタッチメントポイント（空のGameObject）だけは用意しておき、後から拡張しやすい設計にしておきます。
